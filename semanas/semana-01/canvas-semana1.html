Esta semana establece los cimientos conceptuales para todo el curso de SGBD. Estudiaremos los fundamentos de los Sistemas de Gestión de Bases de Datos, desde su definición como interfaz entre datos, usuarios y aplicaciones, hasta la comparación práctica entre modelos relacionales y NoSQL en contextos empresariales reales.

¿Qué veremos?

Tema Contenido clave Aplicaciones prácticas
Definición y Propósito SGBD como colección de datos relacionados y programas que los gestionan. Objetivo: fungir como interfaz entre base de datos, usuarios y aplicaciones para almacenamiento y recuperación flexible de información. Evolución desde sistemas de archivos; gestión empresarial moderna; aplicaciones web y móviles.
Arquitectura ANSI-SPARC Arquitectura de tres niveles propuesta por ANSI-SPARC en 1975: Nivel Interno/Físico (almacenamiento real), Nivel Lógico/Conceptual (estructura completa), Nivel Externo/Vistas (acceso usuarios). Independencia de datos. Separación programas de BD física; vistas personalizadas por departamento; flexibilidad arquitectónica.
Componentes Principales SGBD como paquete de software complejo: Usuarios/Aplicaciones, Interfaz de Usuario, Procesador de Consultas, Motor de BD, Base de Datos, Administrador DBA, Herramientas de Administración trabajando coordinadamente. Diagramas de arquitectura empresarial; mapeo de componentes en sistemas reales; herramientas como phpMyAdmin.
Funciones Principales Funciones esenciales: Gestión de Datos (almacenamiento físico, archivos, índices), Procesamiento de Consultas (análisis SQL, optimización), Control y Seguridad (autenticación, integridad referencial), Administración (backup, monitoreo, mantenimiento). Optimización automática de consultas; sistemas de respaldo empresariales; control de acceso por roles; auditoría.
Arquitectura Cliente/Servidor Modelo de distribución básico y detallado. Cliente (aplicaciones, herramientas, interfaces) conectándose a través de red a Servidor (motor SGBD, procesamiento, transacciones, concurrencia) que accede a Base de Datos. Aplicaciones web/móviles conectándose a servidores BD; arquitecturas distribuidas; escalabilidad empresarial.
Modelo Relacional Postulado por Edgar Frank Codd (1970). Estructura tabular rígida con esquema fijo. Características transaccionales que garantizan integridad y confiabilidad. SQL estándar. Oracle Database, PostgreSQL, MySQL; sistemas bancarios, ERP, CRM; transacciones financieras críticas.
Modelo Documental (NoSQL) Esquema dinámico y flexible. Enfoque en alta disponibilidad, estado adaptable y consistencia gradual. Escalabilidad horizontal. Documentos JSON/XML. MongoDB, CouchDB, Amazon DynamoDB; Big Data, IoT, redes sociales; aplicaciones de alta disponibilidad.
Comparación de Modelos Análisis comparativo detallado: Estructura (tablas vs documentos), Esquema (fijo vs flexible), Escalabilidad (vertical vs horizontal), Consistencia (inmediata vs gradual), Casos de uso específicos según contexto empresarial. Tabla comparativa interactiva; evaluación cuándo usar SQL vs NoSQL; sistemas híbridos modernos; casos de estudio reales.

Objetivos de aprendizaje

Al finalizar la semana podrás:

  • Explicar los conceptos fundamentales de SGBD y su importancia como interfaz entre datos, usuarios y aplicaciones en la gestión moderna de información.

  • Distinguir los tres niveles de la arquitectura ANSI-SPARC y sus beneficios de independencia de datos física y lógica.

  • Identificar y mapear los componentes principales de un SGBD en arquitecturas cliente/servidor, aplicando conceptos a diagramas técnicos reales.
  • Comparar modelos relacionales y documentales, evaluando características de consistencia según el contexto empresarial específico.
  • Analizar casos de uso específicos para determinar cuándo usar SQL vs NoSQL, considerando factores como consistencia, disponibilidad y escalabilidad.
