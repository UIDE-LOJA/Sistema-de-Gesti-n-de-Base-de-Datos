1. Arquitectura de SGBD

1.1 Definición y Propósito

Un Sistema Gestor de Bases de Datos (SGBD) es una colección de datos relacionados entre sí, estructurados y organizados, y un conjunto de programas que acceden y gestionan esos datos.

Objetivo Principal

Fungir como interfaz entre la base de datos, los usuarios y las aplicaciones, proporcionando el método de organización necesario para el almacenamiento y la recuperación flexible de grandes cantidades de información.

1.2 Arquitectura de Tres Niveles (ANSI-SPARC)

La arquitectura más aceptada fue propuesta por el comité ANSI-SPARC en 1975, con el objetivo de separar los programas de aplicación de la BD física.

Navega verticalmente (↓) para explorar cada nivel

Nivel Interno o Físico

Nivel Interno o Físico

Es el nivel más bajo de abstracción y describe cómo se almacenan realmente los datos.

  • Emplea un modelo físico de datos
  • Es el único nivel donde los datos existen realmente
  • Gestiona estructuras de almacenamiento físico
  • Controla índices, archivos y métodos de acceso

Nivel Lógico o Conceptual

Nivel Lógico o Conceptual

Describe la estructura de toda la base de datos para una comunidad de usuarios.

  • Describe entidades, atributos, relaciones
  • Define operaciones y restricciones
  • Oculta detalles de estructuras físicas
  • Asegura consistencia transaccional

Nivel Externo o de Vistas

Nivel Externo o de Vistas

Describe la parte de la BD a la que los usuarios pueden acceder.

  • Puede haber varios esquemas externos
  • Cada vista describe la visión de un grupo de usuarios
  • Oculta el resto de la base de datos
  • Proporciona seguridad y personalización

1.3 Independencia de Datos

Beneficios de la Arquitectura de Tres Niveles

Independencia Lógica

Los cambios en objetos de BD no afectan programas y usuarios

Correspondencia

El SGBD transforma peticiones y resultados entre niveles

2. Componentes del SGBD

2.1 Componentes Generales

Un SGBD es un paquete de software muy complejo compuesto por varios subsistemas para manejar la definición, manipulación, control y generación de aplicaciones.

Motor de la Base de Datos

Motor de la Base de Datos

Componente central que:

  • Acepta peticiones lógicas de otros subsistemas
  • Convierte a su equivalente físico
  • Accede a la base de datos y al diccionario

Procesador del Lenguaje

Intérprete o Procesador del Lenguaje

Responsable de:

  • Procesar sentencias DDL (Lenguaje de Definición de Datos)
  • Procesar sentencias DML (Lenguaje de Manipulación de Datos)
  • Validar sintaxis y semántica

Optimizador y Mecanismos

Optimizador de Consultas

Realiza la optimización de cada pregunta y elige el plan de actuación más eficiente

Mecanismo de Almacenamiento

Traduce operaciones a lenguaje de bajo nivel para acceder a los datos

Motor de Transacciones

Gestiona la ejecución de transacciones para asegurar la corrección y fiabilidad

2.3 Componentes de Memoria (Oracle)

Estructuras de Memoria Principales

En sistemas como Oracle, la arquitectura incluye el Área Global del Sistema (SGA) y el Área Global del Programa (PGA).

Búfer Caché de la Base de Datos

Búfer Caché de la Base de Datos

"Guarda temporalmente los datos más recientemente consultados por los usuarios de base de datos"

  • Mejora el rendimiento de consultas
  • Reduce accesos a disco
  • Implementa algoritmos de reemplazo (LRU)

Búfer de Registro de Operaciones

Búfer de Registro de Operaciones (Redo Log Buffer)

"Guarda un registro de las modificaciones realizadas sobre la base de datos"

  • Asegura la durabilidad de transacciones
  • Permite recuperación ante fallos
  • Registra cambios antes de confirmarlos

Memoria Compartida

Memoria Compartida (Shared Pool)

Guarda temporalmente las sentencias SQL más usadas por los usuarios de la base de datos

  • Cachea planes de ejecución
  • Reutiliza consultas compiladas
  • Mejora eficiencia del sistema

2.4 Procesos de Fondo

Los procesos de fondo mantienen la relación entre las estructuras físicas y de memoria.

Database Writer (DBWR)

Escribe en archivos de datos los bloques modificados

Log Writer (LGWR)

Escribe en disco las entradas del registro

System Monitor (SMON)

Recuperación de instancias durante el inicio

Process Monitor (PMON)

Recupera procesos cuando falla un usuario

3. Modelos de Datos

3.1 Clasificación General

Los modelos de datos definen cómo se estructuran, almacenan y manipulan los datos en un sistema de gestión de bases de datos.

Relacional

Estructura Tabular

SQL, ACID, Esquema Fijo

Documental

Documentos JSON/XML

NoSQL, BASE, Esquema Flexible

3.2 Modelo Relacional

Postulado por Edgar Frank Codd en 1970, basado en la lógica de predicados y la teoría de conjuntos. Es el modelo más utilizado en la actualidad.

Características Principales:

Estructura Tabular Rígida

La información se organiza en relaciones o tablas, compuestas por filas (registros) y columnas (atributos)

Esquema Fijo

Trabajan con un esquema predefinido y estricto que debe respetarse para garantizar la integridad

Lenguaje Estándar

Utiliza SQL (Structured Query Language) para consultar y manipular los datos

Propiedades ACID

Las bases de datos relacionales manejan un conjunto de propiedades definidas por el acrónimo ACID, cruciales para garantizar la confiabilidad.

Atomicidad

Todo o nada

Consistencia

Estado válido

Aislamiento

Transacciones independientes

Durabilidad

Persistencia garantizada

3.3 Modelo Documental (NoSQL)

El término NoSQL ("no solo SQL") se refiere a sistemas de gestión de bases de datos no relacionales que surgieron como alternativa al SGBDR tradicional.

Características Principales:

Esquema Dinámico

No requieren esquemas fijos ni rígidos, permitiendo mayor flexibilidad

Escalabilidad Horizontal

Diseñadas para escalar horizontalmente usando clusters distribuidos

Almacenamiento de Documentos

Los datos se almacenan en documentos JSON/XML con estructura flexible

Propiedades BASE

Los sistemas NoSQL siguen las propiedades BASE en lugar de ACID, priorizando disponibilidad y tolerancia de partición.

Basically Available

El sistema está básicamente disponible la mayor parte del tiempo

Soft State

El estado del sistema puede cambiar con el tiempo

Eventual Consistency

Los datos serán consistentes eventualmente, no inmediatamente

3.4 Comparación: Relacional vs Documental

Aspecto Modelo Relacional Modelo Documental
Estructura Tablas con filas y columnas Documentos JSON/XML
Esquema Fijo y predefinido Flexible y dinámico
Escalabilidad Vertical (más potencia) Horizontal (más nodos)
Consistencia ACID - Inmediata BASE - Eventual
Casos de Uso Transacciones financieras, ERP Big Data, IoT, redes sociales

  • Los componentes del SGBD trabajan coordinadamente para gestionar datos eficientemente
  • Los modelos relacionales y documentales tienen aplicaciones específicas según el contexto
  • La elección del modelo depende de requisitos de consistencia vs disponibilidad

  • Proporcionar acceso concurrente y seguro
  • Adaptar la arquitectura a las necesidades específicas
