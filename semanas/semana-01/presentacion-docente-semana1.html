Contexto de la Problemática

El Desafío de la Gestión de Información

En el contexto digital actual, las organizaciones enfrentan un crecimiento exponencial en el volumen de datos generados por:

  • Aplicaciones de streaming y entretenimiento digital
  • Plataformas de comercio electrónico
  • Redes sociales y sistemas de comunicación
  • Sistemas transaccionales empresariales

Pregunta fundamental: ¿Cómo gestionar eficientemente este universo masivo de información de manera rápida, segura y confiable?

Sistema Gestor de Bases de Datos

Definición Formal

Un Sistema Gestor de Bases de Datos (SGBD) es una colección de datos relacionados entre sí, estructurados y organizados, junto con un conjunto de programas que acceden y gestionan esos datos.

Función Principal

Actúa como interfaz inteligente entre:

  • La base de datos (almacenamiento físico)
  • Los usuarios (personas y aplicaciones)
  • Las aplicaciones (sistemas que requieren datos)

Función Técnica del SGBD

Gestión de Complejidad

Función técnica: Gestiona la complejidad del almacenamiento y recuperación de información, proporcionando una capa de abstracción entre las aplicaciones y el almacenamiento físico de datos.

Beneficios de la Abstracción

  • Simplifica el acceso a datos para desarrolladores
  • Oculta la complejidad del almacenamiento físico
  • Proporciona interfaces estándar (SQL, APIs)
  • Permite optimizaciones transparentes

Arquitectura ANSI-SPARC de Tres Niveles

Fundamento Arquitectónico (1975)

En 1971, el DBTG (DataBase Task Group) reconoció la necesidad de un enfoque de dos niveles con vistas y esquemas. Posteriormente, en 1975, ANSI-SPARC desarrolló un enfoque de tres niveles con niveles de abstracción: externo, conceptual e interno.

El objetivo fundamental es separar la vista de cada usuario de la base de datos de la forma como está físicamente representada.

Principio Clave

Independencia de datos: Permite modificar la infraestructura física sin afectar las aplicaciones que utilizan los datos.

Beneficios Fundamentales

  • Independencia Lógica: Cambios en el esquema conceptual no afectan programas de aplicación
  • Independencia Física: Cambios en almacenamiento no impactan el nivel conceptual
  • Correspondencia: SGBD transforma peticiones entre niveles automáticamente
  • Flexibilidad: Modificaciones transparentes para usuarios

Niveles de Abstracción Detallados

Nivel Externo

Vista del usuario: Describe cómo cada usuario ve la base de datos.

  • Contiene solo entidades, atributos y relaciones relevantes para el usuario
  • Diferentes vistas pueden representar los mismos datos de maneras distintas
  • Ejemplo: nombre como (apellido, nombre) vs (nombre, apellido)
  • Múltiples vistas externas por base de datos

Nivel Conceptual

Vista de la comunidad: Describe qué datos se almacenan y sus relaciones.

  • Contiene la estructura lógica completa de toda la base de datos
  • Representa entidades, atributos, relaciones, operaciones de usuario
  • Incluye información semántica, de seguridad e integridad
  • Independiente de consideraciones de almacenamiento

Nivel Interno

Representación física: Cómo la base de datos se representa físicamente en el computador.

  • Enfatiza la implementación física para optimización de espacio y rendimiento
  • Utiliza estructuras de datos complejas para optimización
  • Se interfaza con el sistema operativo para almacenamiento y recuperación
  • Maneja técnicas de encriptación y compresión de datos

Modelos de Datos

Definición

Los modelos de datos definen cómo se estructuran, almacenan y manipulan los datos en un sistema de gestión de bases de datos.

Modelo Relacional

  • Estructura: Tabular
  • Características: SQL, Transaccional
  • Esquema: Fijo y predefinido

Modelo Documental

  • Estructura: Documentos JSON/XML
  • Características: NoSQL, BASE
  • Esquema: Flexible y dinámico

Modelo Relacional

Fundamento Histórico

Postulado por Edgar Frank Codd en 1970, basado en la lógica de predicados y la teoría de conjuntos. Es el modelo más utilizado en la actualidad.

Características Principales

  • Estructura Tabular Rígida: Información organizada en relaciones (tablas) con filas y columnas
  • Esquema Fijo: Estructura predefinida y estricta para garantizar integridad
  • Lenguaje Estándar: SQL (Structured Query Language) para consultas y manipulación

Características Transaccionales

Las transacciones son operaciones fundamentales que permiten modificar y recuperar datos de manera segura y confiable.

Nota: Los conceptos detallados de transacciones y propiedades ACID se estudiarán en profundidad en la Semana 2.

Modelo Documental (NoSQL)

Contexto de Surgimiento

El término NoSQL ("no solo SQL") se refiere a sistemas no relacionales que surgieron como alternativa para manejar grandes volúmenes de datos y escalabilidad requerida por aplicaciones web modernas.

Características Principales

  • Esquema Dinámico: No requiere esquemas fijos, mayor flexibilidad
  • Escalabilidad Horizontal: Distribución entre múltiples nodos
  • Almacenamiento de Documentos: Datos en formato JSON/XML

Enfoque de Consistencia

Priorizan disponibilidad y escalabilidad sobre consistencia inmediata:

  • Alta Disponibilidad: Sistema disponible la mayor parte del tiempo
  • Flexibilidad: Estructura adaptable a cambios
  • Consistencia Gradual: Los datos se sincronizan progresivamente

Nota: Los modelos de consistencia detallados se estudiarán en la Semana 2.

Comparación: Relacional vs Documental

Análisis Comparativo

La elección entre modelos depende de los requisitos específicos del proyecto y las características del dominio de aplicación.

Casos de Uso: Modelo Relacional

Contextos Ideales

El modelo relacional es perfecto para sistemas donde la integridad de datos es crítica y se requiere consistencia inmediata.

Sistemas Financieros

  • Transacciones bancarias
  • Sistemas de pago
  • Contabilidad empresarial
  • Auditoría financiera

Sistemas Empresariales

  • ERP (Enterprise Resource Planning)
  • CRM (Customer Relationship Management)
  • Sistemas de inventario
  • Gestión de recursos humanos

Ejemplos de Tecnologías

Oracle Database, PostgreSQL, MySQL, Microsoft SQL Server

Casos de Uso: Modelo Documental

Contextos Ideales

El modelo documental brilla en entornos que requieren flexibilidad y escalabilidad masiva con esquemas dinámicos.

Big Data y Analytics

  • Análisis de grandes volúmenes
  • Data warehousing
  • Business intelligence
  • Machine learning datasets

Aplicaciones Modernas

  • Redes sociales
  • Internet de las cosas (IoT)
  • Aplicaciones móviles
  • Sistemas de recomendación

Ejemplos de Tecnologías

MongoDB, CouchDB, Amazon DynamoDB, RavenDB

Tendencias y Perspectivas Futuras

Evolución del Panorama de Datos

En un contexto de datos cada vez más complejos e interconectados, surge la pregunta: ¿dominará un solo modelo o la clave será la coexistencia?

Enfoque Híbrido

Las evidencias apuntan hacia un futuro híbrido donde diferentes arquitecturas colaboran según las necesidades específicas:

  • Sistemas poliglotas que combinan múltiples modelos
  • Arquitecturas de microservicios con bases de datos especializadas
  • Integración de modelos relacionales y NoSQL

Competencia Profesional

La habilidad clave será saber elegir el modelo arquitectónico adecuado para cada tipo de construcción de sistema de información.
