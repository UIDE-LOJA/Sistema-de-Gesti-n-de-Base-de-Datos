Esta semana profundizamos en los conceptos fundamentales de transacciones y las propiedades ACID que garantizan la integridad de datos. Exploraremos técnicas avanzadas de control de concurrencia y protocolos de bloqueo, fundamentales para sistemas multiusuario que requieren procesamiento transaccional confiable y eficiente.

¿Qué veremos?

Tema Contenido clave Ejemplos
Conceptos de Transacciones Unidad lógica de trabajo que transforma la base de datos de un estado consistente a otro. Estados del ciclo de vida: activa, parcialmente confirmada, confirmada, fallida o abortada. Operaciones fundamentales: BEGIN, READ, WRITE, COMMIT, ROLLBACK. Transferencia bancaria (débito + crédito atómico); reserva de vuelo con asiento, pago y confirmación; actualización de inventario en e-commerce.
Propiedades ACID Atomicidad: Todo o nada - transacción completa o rollback total. Consistencia: Mantiene reglas de integridad y restricciones. Aislamiento: Transacciones concurrentes no se interfieren. Durabilidad: Cambios confirmados persisten ante fallos del sistema. Falla en transferencia → rollback automático; integridad referencial preservada; logs de transacciones para recuperación.
Niveles de Aislamiento Jerarquía de control de visibilidad: SERIALIZABLE (máxima consistencia), REPEATABLE READ (lecturas consistentes), READ COMMITTED (datos confirmados), READ UNCOMMITTED (máximo rendimiento). Trade-off consistencia vs. concurrencia. Sistemas bancarios (SERIALIZABLE); reportes analíticos (READ COMMITTED); dashboards en tiempo real (READ UNCOMMITTED).
Técnicas de Bloqueo Protocolos de bloqueo: Compartido (S) para lecturas, Exclusivo (X) para escrituras, Actualización (U) para modificaciones. Granularidad variable: fila, página, tabla. Two-Phase Locking (2PL) garantiza serializabilidad. SELECT con lock compartido; UPDATE con lock exclusivo; detección y resolución de deadlocks; timeout de transacciones.
MVCC vs Bloqueo Multi-Version Concurrency Control (MVCC): Múltiples versiones de datos para lecturas sin bloqueo. Timestamp Ordering: Ordenamiento temporal para serializabilidad. Comparación con protocolos de bloqueo tradicionales y sus trade-offs de rendimiento. PostgreSQL MVCC; Oracle Flashback; SQL Server snapshot isolation; sistemas de alta concurrencia sin deadlocks.
Fenómenos de Concurrencia Dirty Read: Lectura de datos no confirmados. Non-repeatable Read: Valor cambia entre lecturas de la misma transacción. Phantom Read: Nuevas filas aparecen en consultas repetidas. Deadlocks: Detección por grafos de espera y resolución automática. Simulación con dos sesiones SQL; análisis de logs de transacciones; configuración de timeouts; estrategias de prevención.

Objetivos de aprendizaje

Al finalizar la semana podrás:

  • Definir transacciones y explicar las propiedades ACID, analizando su importancia crítica en sistemas transaccionales (OLTP) y procesamiento confiable de datos.

  • Demostrar control de concurrencia en SGBD mediante técnicas de bloqueo, protocolos 2PL y comparación con enfoques MVCC para sistemas multiusuario.

  • Diseñar pruebas para verificar propiedades ACID y simular fenómenos de concurrencia (dirty reads, non-repeatable reads, phantom reads) usando sesiones SQL paralelas.
  • Seleccionar niveles de aislamiento apropiados para diferentes escenarios empresariales, justificando el balance entre consistencia, rendimiento y concurrencia.
