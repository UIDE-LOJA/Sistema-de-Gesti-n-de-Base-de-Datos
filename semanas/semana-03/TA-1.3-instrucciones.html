Conceptos clave a incluir:

  • ARIES en InnoDB: Implementación específica de MySQL para recuperación ante fallos
  • LSN (Log Sequence Number): Identificador único de cada cambio en InnoDB
  • Redo Log: Registro físico de cambios (WAL - Write-Ahead Logging)
  • Undo Log: Registro para revertir transacciones no confirmadas
  • Buffer Pool: Dirty pages con LSN para recuperación
  • Checkpoint: Punto de referencia para iniciar recuperación

Ejemplo de esquema ARIES:

CRASH → RECOVERY PROCESS

1. ANALYSIS
   - Leer log desde último checkpoint
   - Identificar transacciones activas
   - Identificar páginas sucias
   ↓
2. REDO
   - Repetir TODAS las operaciones
   - Restaurar estado al momento del crash
   ↓
3. UNDO
   - Revertir transacciones sin COMMIT
   - Restaurar consistencia
   ↓
RECOVERY COMPLETE

Ejemplo aplicado al Sistema de Pagos QR:

Escenario: Servidor MySQL crashea durante procesamiento de pagos

Antes del crash:
- T1: Pago Edgar en Cafetería UIDE $2.50 → COMMIT ✓ (confirmada)
- T2: Pago Anyela en Farmacia $3.75 → SIN COMMIT ✗ (incompleta)
- CRASH! ⚡

Recuperación ARIES en InnoDB:
1. ANALYSIS:
   - Leer desde último checkpoint (LSN: 123450000)
   - Identificar: T1 confirmada, T2 activa sin COMMIT
   - Reconstruir Transaction Table y Dirty Page Table

2. REDO:
   - Rehacer AMBAS transacciones desde checkpoint
   - T1: Débito cuenta 101 (-$2.50), Crédito cuenta 202 (+$2.50)
   - T2: Débito cuenta 103 (-$3.75), Crédito cuenta 203 (+$3.75)
   - Estado restaurado al momento exacto del crash

3. UNDO:
   - Revertir T2 (no confirmada):
     • Restaurar saldo cuenta 103 (+$3.75)
     • Restaurar saldo cuenta 203 (-$3.75)
   - T1 permanece confirmada

Resultado final:
✓ Pago de Edgar persiste (durabilidad ACID)
✗ Pago de Anyela se revierte (atomicidad ACID)
✓ Sistema consistente y recuperado
